# 营销导购陪练智能体 · 作品集发布指南

本文档说明如何将本作品集发布到 **GitHub 仓库** 或 **飞书文档** 两种渠道，并对比两种方案的优缺点，帮助你选择最适合的发布方式。

---

## 方案 A：发布到 GitHub

### 适用场景
- 希望作品集可被技术 / 产品招聘方直接检索、克隆、查看源码级细节
- 需要长期沉淀为个人技术品牌资产
- 6 份原型 HTML 希望在线可交互预览（GitHub Pages）

### 完整步骤

**1. 初始化本地仓库**
```bash
cd "F:\Marvis\营销导购陪练智能体-作品集发布"
git init
git add .
git commit -m "feat: 初始化营销导购陪练智能体作品集"
```

**2. 在 GitHub 创建远程仓库**
- 登录 GitHub → New repository → 命名为 `sales-training-agent-portfolio`
- 不要勾选 Initialize with README（本地已有）
- 复制仓库 HTTPS / SSH 地址

**3. 关联远程并推送**
```bash
git branch -M main
git remote add origin https://github.com/USERNAME/sales-training-agent-portfolio.git
git push -u origin main
```

**4. 开启 GitHub Pages 预览原型**
- 进入仓库 **Settings → Pages**
- Source 选择 `main` 分支、`/ (root)` 目录 → Save
- 等待 1–2 分钟，访问如下地址即可在线交互预览原型（将 `USERNAME` / `REPO` 替换为实际值）：
  - `https://USERNAME.github.io/REPO/prototype/营销导购陪练智能体-交互原型.html`
  - `https://USERNAME.github.io/REPO/prototype/营销导购陪练智能体-学员端.html`
  - `https://USERNAME.github.io/REPO/prototype/营销导购陪练智能体-管理端.html`
  - `https://USERNAME.github.io/REPO/prototype/营销导购陪练智能体-双端原型.html`
  - `https://USERNAME.github.io/REPO/prototype/语音陪练原型-营销导购陪练智能体.html`
  - `https://USERNAME.github.io/REPO/prototype/产品路线图-营销导购陪练智能体.html`

> 提示：中文文件名在部分浏览器地址栏会被 URL 编码（如 `%E8%90%A5...`），复制 README 中的相对链接点击访问即可，无需手动转码。

**5. 完善仓库信息**
- 在仓库 About 中填写简介、添加 Topics（如 `product-management`、`ai-agent`、`sales-training`、`saas`、`prototype`）
- README.md 已使用相对路径（`docs/xxx.md`、`prototype/xxx.html`），推送后链接自动生效

### 优点
- 链接永久稳定、可被搜索引擎收录
- 6 份原型 HTML 通过 GitHub Pages 在线可交互，体验最佳
- 支持版本管理，便于持续更新作品集
- 招聘方（尤其技术型 PM / 大厂）认可度高

### 缺点
- 需要一定 Git 操作基础
- 飞书 / 微信等国内平台内打开 GitHub 链接可能受限（需科学上网）
- 文档排版依赖 Markdown 渲染，复杂表格 / 图片需额外处理
- 中文文件名的 Pages 链接会被 URL 编码，观感略长

## 方案 B：发布到飞书文档

### 适用场景
- 主要面向国内招聘方、内推渠道、飞书生态企业
- 希望文档可评论、可协作、可嵌入飞书多维表格 / 云文档
- 不需要源码级托管，重在内容呈现

### 完整步骤

**1. 创建飞书文档**
- 打开飞书 → 新建「文档」→ 命名为「营销导购陪练智能体 · 项目作品集」

**2. 粘贴 README 内容**
- 将本目录 `README.md` 内容复制粘贴进飞书文档
- 飞书支持 Markdown 粘贴，标题 / 表格 / 列表会自动格式化
- 顶部 Shields.io badge 图片链接可直接保留（飞书会渲染为图片）

**3. 插入原型截图与演示**
- 用浏览器打开 `prototype/` 下 6 个原型 HTML
- 对关键页面分别截图：学员端（工作台 / 文字陪练 / 5 维评估报告 / 对话回放）、管理端（团队看板 / 场景管理 / RBAC 矩阵）、语音陪练（打断指示灯 / 通话计时）
- 在飞书文档对应章节插入截图，并附一句说明 + 原型文件下载链接（可上传到飞书云文档）
- 双端原型可录制一段"一键切换视角"的短视频嵌入，最直观展示双角色协同

**4. 补充文档索引**
- 将 `docs/` 下 10 个 Markdown 文档逐一上传为飞书云文档，或在主文档中以「链接卡片」形式引用
- 重点突出 PRD（14 章）、一页纸摘要、竞品分析摘要表、简历项目经历四份
- 也可将整个作品集文件夹上传到飞书云空间，生成共享链接

**5. 设置权限与分享**
- 点击「分享」→ 设置「获得链接的人可阅读」
- 复制链接，附在简历 / 内推消息中

### 优点
- 国内访问无障碍，招聘方打开即用
- 支持评论、协作、@ 提醒，便于面试前沟通
- 可嵌入图片 / 视频 / 云文档，呈现形式丰富
- 无需技术基础，上手快

### 缺点
- 链接依赖飞书账号体系，离职 / 注销后可能失效
- 原型 HTML 无法在飞书内直接交互，只能以截图 / 视频 / 下载形式呈现
- 不利于搜索引擎收录与个人技术品牌长期沉淀
- 文档版本管理弱于 Git

## 两种方案优缺点对比

| 对比维度 | 方案 A：GitHub | 方案 B：飞书文档 |
|----------|----------------|------------------|
| 国内访问 | 可能受限（需科学上网） | 无障碍 |
| 原型交互 | ✅ GitHub Pages 在线可交互 | ❌ 仅截图 / 视频 / 下载 |
| 搜索引擎收录 | ✅ 可被收录 | ❌ 封闭生态 |
| 版本管理 | ✅ Git 完整历史 | ⚠️ 弱 |
| 协作评论 | ⚠️ 仅 Issue / PR | ✅ 原生评论 @ |
| 技术品牌沉淀 | ✅ 强 | ⚠️ 弱（依赖账号） |
| 操作门槛 | 中（需 Git 基础） | 低（复制粘贴） |
| 招聘方认可度 | 技术型 / 大厂高 | 国内通用型高 |

### 推荐策略
- **技术导向 / 海外 / 大厂岗位**：优先方案 A，并将 GitHub 链接附在简历显眼位置。
- **国内通用 / 内推为主**：优先方案 B，或将方案 A 的 README 同步发布到飞书。
- **最佳实践**：GitHub 托管完整作品集（含 6 份可交互原型）+ 飞书文档做精简版投递版，双渠道覆盖。
