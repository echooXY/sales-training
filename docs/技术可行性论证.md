# 技术可行性论证

---

## 1. 核心架构概述

PRD中定义的营销导购陪练智能体底层架构遵循 `Agent = Model, Harness = Code` 工程哲学——Agent负责智力（LLM推理），Harness负责给模型装工具、管记忆、建团队、做评估。系统分为三层：

```
前端（React）：角色配置面板 / 陪练对话界面（文字+语音） / 评估报告与看板 / 管理后台
    ↓ REST API / WebSocket / WebRTC
Training Backend（Python微服务）：Lead Agent Loop + Tool Handlers + MessageBus + TaskSystem + Voice Engine
    ↓
Infrastructure：PostgreSQL（多租户） + Redis（消息队列） + LLM API（Claude/GPT/千问/文心） + WebRTC SFU
```

这套架构在技术文档的七章中已有完整的代码级实现描述，覆盖Lead Agent主循环、MessageBus JSONL通信、三层上下文压缩、评估子Agent、双层进度追踪等核心模块。以下逐模块论证可行性。

---

## 2. 关键模块可行性

### 2.1 多Agent编排（Lead + Customer + Evaluator）

**PRD需求**：学员扮演销售与多个AI顾客对话，顾客之间通过MessageBus JSONL邮箱独立推理并互相协商，模拟"夫妻/朋友/同事商量"效果。

**技术实现**：
- `spawn_teammate` 工具启动独立线程，每个Customer Agent拥有独立的 `messages[]` 和角色Prompt，通过JSONL文件（`.team/` 目录）实现异步邮箱通信。
- Lead Agent主循环每轮读取MessageBus收件箱 → LLM推理 → 执行tool调用 → 将回复写入发件箱推送至Customer Agent。
- Customer Agent独立线程并行处理，各自调用LLM生成回复后写入MessageBus。

**可行性判定：✅ 可落地**

**论证**：多线程+JSONL通信是成熟的分布式通信模式，技术文档中已有完整的时序图实现描述。核心风险在于3+顾客场景每轮12-20次API调用的延迟和成本，PRD已设计了30秒超时降级和MVP阶段并发限制作为缓解措施。

---

### 2.2 Prompt即场景的边界与保底

**PRD需求**：企业培训经理仅需编写角色Prompt即可在30分钟内创建新场景，无需预标注对话数据。

**技术实现**：
- 角色Prompt分为三层结构：身份描述（角色背景/性格/目标）+ 行为约束（对话风格/禁忌）+ 评估标准（衡量对话质量的目标描述）。
- 场景配置保存为文件，Lead Agent通过 `load_skill` 按需加载。
- 无需模型微调或RAG检索标注对话库。

**可行性判定：✅ 可落地**

**论证**：PRD第4章定义了四级难度体系（L1友好型→L4多人决策），本质是Prompt复杂度递增而非架构切换。当Prompt不足以覆盖严格合规场景时，PRD第12章风险评估已识别可通过规则引擎叠加SOP匹配模块扩展，属于有条件可落地的延伸。

---

### 2.3 三层上下文压缩

**PRD需求**：30-80轮对话后token不超过80K，压缩后角色人格不丢失。

**技术实现**：

| 层级 | 触发条件 | 动作 | 技术原理 |
|------|---------|------|---------|
| microcompact | 每轮LLM调用前 | 清理3轮前的tool_result→[cleared] | 零开销字符串替换 |
| auto_compact | 估算token超过阈值 | LLM生成延续性摘要，完整对话存为 `.transcripts/` JSONL | 结构化摘要+完整归档 |
| manual | 场景切换/主动触发 | Lead Agent自主决策触发时机 | 事件驱动 |

**身份重注入**：压缩后Lead Agent检测到压缩事件，自动将角色Agent的identity prompt重新注入消息列表。

**可行性判定：✅ 可落地**

**论证**：microcompact是纯工程优化（字符串替换），auto_compact依赖LLM生成本身具备的摘要能力，身份重注入是消息列表操作。三层策略已在技术文档第4章有完整实现描述，其中microcompact和auto_compact已在Agent Loop主循环中集成。

---

### 2.4 独立评估子Agent

**PRD需求**：陪练结束后独立派生子Agent，以只读模式分析完整对话历史，按5维16粒度逐项打分，评估偏差≤5分（百分制）。

**技术实现**：
- 评估子Agent通过 `evaluate` 工具触发，派生出独立的 `messages[]`，只读访问对话历史和评估量表，禁止写入/修改操作。
- 按结构化评估量表（维度权重 + 子评分点 + 0-10分锚定描述）逐维度分析后输出JSON格式评分。
- 评估完成后messages[]丢弃，不污染主Agent上下文。

**可行性判定：✅ 可落地**

**论证**：子Agent派生的核心是创建新的LLM上下文并注入评估指令，技术难度低。质量保障的关键在于评估量表的锚定描述精度和人工双评校准（PRD SMART目标：偏差≤5分），这属于产品设计问题而非技术瓶颈。

---

### 2.5 语音接入（ASR + TTS + 打断）

**PRD需求**：端到端延迟≤800ms（ASR 200ms + Agent推理 300ms + TTS 300ms），支持PTT/VAD/打断三种交互模式。

**技术实现**：
- WebRTC SFU建立实时音视频通道，前端采集音频 → 后端ASR引擎转录文本 → 文本进入Lead Agent Loop → 回复文本 → TTS引擎合成语音 → 前端播放。
- 打断模式：VAD检测到学员语音时，300ms内中断TTS播放并切换到收听状态。
- 多路ASR引擎热备（阿里云+讯飞+Whisper）+ 行业热词表定制提升准确率。
- TTS多声线支持（男/女/老年/年轻），MOS≥4.0。

**可行性判定：✅ 可落地**

**论证**：ASR和TTS均为成熟云服务（阿里云/讯飞/Whisper），WebRTC是成熟的实时通信协议。800ms延迟目标的关键瓶颈在Agent推理（300ms），取决于LLM响应速度——技术文档中已有备用模型路由降级策略（如Claude→GPT-4o）。降级设计完备（ASR连续失败→切换文字、WebRTC断连→3秒自动重连、网络抖动→降低TTS码率）。

---

### 2.6 私有化部署

**PRD需求**：支持Docker Compose / K8s部署，LLM对接企业自有模型（千问/文心），语音引擎支持离线ASR/TTS（PaddleSpeech），数据不出内网。

**技术实现**：
- 系统三层架构均为容器化部署设计：React前端（Nginx静态服务）、Python Backend微服务、PostgreSQL+Redis基础设施。
- LLM API通过环境变量注入，兼容OpenAI API格式，支持切换任何兼容模型。
- 离线ASR/TTS使用PaddleSpeech等开源引擎替代云服务。

**可行性判定：⚠️ 有条件可落地**

**论证**：容器化部署是成熟方案。主要条件在两个方面：①离线ASR/TTS的准确度和自然度可能低于云服务（PaddleSpeech vs 阿里云/讯飞），需在私有化部署SLA中说明性能差异；②LLM推理需企业自备GPU算力或对接企业已有模型API，部分中小企业可能不满足硬件门槛。PRD定价中私有化部署按年授权费+一次性部署实施费的模式是合理的。

## 3. 技术风险与缓解

| 风险 | 等级 | 缓解措施 | 残留风险 |
|------|------|---------|---------|
| LLM API延迟波动 | 中 | 备用模型路由降级、前端加载状态 | 高峰期仍可能超3秒 |
| 上下文压缩后角色人格丢失 | 低 | 身份重注入+压缩后自动验证 | 极端长对话（100+轮）待验证 |
| 多Agent并发通信超时 | 中 | 30秒超时+预设回复降级 | 降级回复可能略显生硬 |
| 语音模式ASR准确率行业差异大 | 中 | 多路引擎热备+行业热词表 | 特定方言/口音场景待优化 |
| 3+顾客并发API调用成本高 | 中 | MVP限制并发+Phase 1优化批处理 | L4高并发场景成本待实测 |
| 企业私有化部署GPU资源门槛 | 低 | 明确硬件要求+提供云部署过渡方案 | 极小企业无法满足 |

**结论**：所有已识别风险均有明确缓解措施，无阻塞性技术风险。

## 4. 与竞品技术方案对比

| 技术维度 | 本产品 | Megaview | FindAI | 得助智能 |
|---------|--------|----------|--------|---------|
| 场景创建 | Prompt驱动（零标注） | 需标注对话库+RAG | 需预置话术库 | 300+预训练智能体 |
| Agent架构 | 多Agent独立LLM推理 | MegaAgents | 单一AI | 多智能体平台 |
| 上下文管理 | 三层压缩+身份重注入 | 未公开 | 未公开 | 未公开 |
| 评估方式 | 独立子Agent+结构化量表 | SOP分析模块 | 四维评估 | 学练考评闭环 |
| 进度追踪 | TodoWrite+TaskSystem DAG | 无 | 无 | 无 |
| 语音模式 | ASR+TTS+打断 | TTS/ASR | 智能耳机硬件 | 内置 |
| 私有化部署 | Docker/K8s+离线引擎 | 不支持 | 不支持 | 支持（金融定制） |

**技术差异化总结**：
- Prompt即场景 + 三层压缩 + 独立评估子Agent + TaskSystem DAG 构成技术护城河
- 多人决策L4的多Agent独立推理能力是目前竞品未覆盖的
- 语音模式采用成熟云服务而非自研ASR/TTS，工程策略务实

---

## 5. 总体结论

**本产品 PRD V1.1 中定义的核心需求全部技术可落地。**

- Must 功能（文字陪练L1-L3 + 5维评估 + 三层压缩 + SSO/RBAC）：**可落地**，技术文档已有代码级实现描述
- Should 功能（L4多人决策 + 语音陪练 + 团队看板）：**可落地**，关键技术组件（多Agent/WebRTC/ASR/TTS）均为成熟技术
- Could 功能（私有化部署 + CRM集成 + API开放平台）：**有条件可落地**，私有化需企业自备GPU或对接自有模型
- Won't 功能（视频陪练 + AI实时弹窗）：远期规划，技术路径清晰但不在当前论证范围
